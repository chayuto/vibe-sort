# v0.4.0 Plan: OpenRouter, Key Detection, and Extra Params

## Executive Summary

This document outlines the plan for VibeSort v0.4.0, which adds three related features:

1. **OpenRouter provider** — a sixth provider adapter, OpenAI-compatible
2. **API key format detection** — infer the provider from the key prefix when `provider:` is omitted; warn (never fail) on mismatch when it is explicit
3. **`extra_params` escape hatch** — a single hash of provider-native request parameters deep-merged into the payload, instead of enumerating every model knob as first-class config

**Status:** Implemented — shipped in v0.4.0

**Target Version:** 0.4.0 (minor bump — provider auto-detection is a behavior change, see Compatibility Notes)

---

## Motivation

- OpenRouter gives access to hundreds of models through one OpenAI-compatible endpoint — maximum flavour for a gem whose whole point is sorting arrays with unreasonable amounts of AI.
- Provider key prefixes are distinctive enough that `VibeSort::Client.new(api_key: key)` can usually "just work" without a `provider:` argument. That fits the gem's vibe.
- Users routing through OpenRouter (or tuning any provider) need occasional knobs (`max_tokens`, `top_p`, OpenRouter `provider` routing preferences). One passthrough hash covers all of them without a config matrix to maintain.

## Non-Goals

- First-class config attributes for every model parameter (`top_p`, `max_tokens`, penalties, …) — `extra_params` covers these
- Validating API keys against provider formats (prefixes are conventions, not contracts — OpenAI already changed theirs once with `sk-proj-`)
- Cross-provider normalization of `extra_params` keys (they are provider-native by design)
- Streaming, retries, or any other transport changes

## Feature 1: OpenRouter Provider

### Design

`Providers::OpenRouter < OpenAI`, mirroring `groq.rb` and `space_x_ai.rb`:

```ruby
module VibeSort
  module Providers
    # Adapter for the OpenRouter API (OpenAI-compatible Chat Completions)
    class OpenRouter < OpenAI
      ENDPOINT = "https://openrouter.ai/api/v1/chat/completions"
      DEFAULT_MODEL = "openai/gpt-4o-mini"

      private

      def provider_name
        "OpenRouter"
      end

      def endpoint
        ENDPOINT
      end

      def headers
        super.merge(
          "HTTP-Referer" => "https://github.com/chayuto/vibe-sort",
          "X-Title" => "vibe-sort"
        )
      end
    end
  end
end
```

### Decisions

- **Default model:** `openai/gpt-4o-mini` for parity with the OpenAI adapter's default. OpenRouter is the provider where users are most likely to pass `model:` anyway (already supported).
- **Attribution headers:** `HTTP-Referer` / `X-Title` are optional but recommended by OpenRouter; they cost nothing and get the gem listed on openrouter.ai rankings.
- **JSON mode caveat:** OpenRouter forwards `response_format: { type: "json_object" }` but only some underlying models honor it. The existing `JSON::ParserError → ApiError` path already handles models that ignore it; document the caveat in the README and note that `extra_params` can override `response_format` if needed.

## Feature 2: Key Format Detection

### Design

A small module with an ordered prefix map (order matters — specific prefixes before `sk-`):

```ruby
module VibeSort
  # Best-effort provider inference from API key prefixes.
  # Prefixes are conventions, not contracts — used only as a default
  # and for non-fatal mismatch warnings, never for validation.
  module KeyDetector
    PREFIXES = {
      "sk-ant-" => :anthropic,
      "sk-or-"  => :openrouter,
      "sk-"     => :openai,
      "gsk_"    => :groq,
      "AIza"    => :gemini, # classic Google API key
      "AQ."     => :gemini, # newer Gemini API key format (observed in the wild, 2026)
      "xai-"    => :spacexai
    }.freeze

    # @return [Symbol, nil] detected provider, or nil if unrecognized
    def self.detect(api_key)
      PREFIXES.each { |prefix, provider| return provider if api_key.start_with?(prefix) }
      nil
    end
  end
end
```

### Wiring in `Configuration`

`provider:` defaults change from `:openai` to `nil` in both `Client` and `Configuration`:

- `provider: nil` (omitted) → `KeyDetector.detect(api_key) || :openai`
- `provider:` explicit and detection disagrees → `Kernel#warn` one line (e.g. `vibe-sort: api_key looks like an anthropic key but provider is :openai`) and proceed with the explicit provider
- `provider:` explicit and detection returns `nil` → silent (unknown formats are normal: proxies, Azure, self-hosted gateways)

Providers ship multiple key formats concurrently (Google: classic `AIza…` and newer `AQ.…`; OpenAI: `sk-…` and `sk-proj-…`), and new formats appear without notice — another reason detection must stay a soft default, never validation. An unrecognized prefix simply means "no inference", not "bad key".

### Behavior Matrix

| `provider:` arg | Key prefix        | Result                          |
|-----------------|-------------------|---------------------------------|
| omitted         | `sk-ant-…`        | `:anthropic` (detected)         |
| omitted         | unrecognized      | `:openai` (unchanged fallback)  |
| `:openai`       | `sk-…`            | `:openai`, no warning           |
| `:openai`       | `sk-ant-…`        | `:openai`, one-line warning     |
| `:groq`         | unrecognized      | `:groq`, no warning             |

## Feature 3: `extra_params` Escape Hatch

### Design

One new optional config attribute, deep-merged into the request payload **last** so it can override anything:

```ruby
# Configuration
def initialize(api_key:, temperature: 0.0, provider: nil, model: nil, extra_params: {})
```

In `Providers::Base`:

```ruby
def perform(array)
  response = connection.post do |req|
    req.body = deep_merge(build_payload(array), config.extra_params)
  end
  handle_response(response)
end
```

with a private `deep_merge` helper (recursive for nested hashes, e.g. Gemini's `generationConfig`; arrays and scalars replace).

### Decisions

- **Provider-native keys, no translation.** `{ max_tokens: 100 }` means what OpenAI/Groq/OpenRouter/xAI say it means; Gemini users write `{ generationConfig: { maxOutputTokens: 100 } }`. Documented explicitly.
- **Merged last** so users can override `response_format`, `temperature`, even `model` — full escape hatch semantics.
- **Symbol keys** in examples; deep_merge treats the hash as-is (payloads are built with symbol keys throughout).

### Checklist

- [x] `Configuration`: `extra_params` attr + default `{}`
- [x] `Client`: passthrough kwarg + doc example
- [x] `Base#perform`: merge step + `deep_merge` helper
- [x] Specs: merge adds new keys, overrides existing (`response_format`), deep-merges nested hashes, empty hash is a no-op (existing payload specs unchanged)
- [x] README: examples — `max_tokens` on OpenAI, `generationConfig` on Gemini, `provider` routing prefs on OpenRouter

## Compatibility Notes

- **Behavior change:** an Anthropic/Gemini/Groq/xAI/OpenRouter-prefixed key with **no** `provider:` argument previously hit the OpenAI API and failed; it now routes to the detected provider and works. Strictly an improvement, but observable — hence 0.4.0, not 0.3.2.
- Unrecognized keys with no `provider:` still default to `:openai` (unchanged).
- All existing explicit-provider call sites behave identically (worst case: a new one-line stderr warning on a mismatched key).
- No new runtime dependencies.

## Implementation Order

1. `extra_params` (self-contained, touches `Base` — land first so the OpenRouter spec can cover interplay)
2. OpenRouter provider
3. Key detection (depends on `:openrouter` existing for the `sk-or-` mapping)
4. Docs sweep: README, CHANGELOG, `docs/api_reference.md`, `docs/architecture.md` provider list
5. Version bump to 0.4.0

Each step is independently green (rubocop + rspec) and committable.
